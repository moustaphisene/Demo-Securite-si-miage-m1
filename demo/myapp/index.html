Flux OAuth2 Authorization Code + PKCE (S256) — Etape par etape

1
Generation du code_verifier (cote client) L'app genere un secret aleatoire (code_verifier) et calcule son hash SHA-256 (code_challenge = SHA256(verifier)).
2
Redirection vers Keycloak L'app redirige l'utilisateur vers Keycloak en envoyant le code_challenge (jamais le secret).
3
Authentification sur la page Keycloak L'utilisateur saisit ses identifiants SUR la page Keycloak. L'app ne voit jamais le mot de passe.
4
Retour avec un authorization_code ephemere Keycloak redirige vers l'app avec un code dans le fragment (#) de l'URL. Ce code est valide ~1 minute, une seule utilisation.
5
Echange du code contre les tokens L'app envoie le code + le code_verifier original. Keycloak verifie que SHA256(verifier) == challenge et retourne les 3 tokens.
Pourquoi PKCE ?
Si un attaquant intercepte le authorization_code (ex: via les logs de proxy), il ne peut rien faire car il n'a pas le code_verifier qui n'a jamais transite sur le reseau. PKCE est obligatoire pour les apps publiques (SPA, mobile) qui n'ont pas de client_secret.

azpAuthorized party — client qui a demande le token

Securite — Bonnes pratiques

A faire cote serveur (API) :
• Verifier la signature avec la cle publique Keycloak
• Verifier exp (non expire)
• Verifier iss (votre realm Keycloak)
• Verifier aud (votre client)
• Verifier les roles requis dans realm_access
A faire cote client (SPA) :
• Stocker les tokens en memoire (pas en localStorage !)
• Utiliser PKCE — obligatoire pour les apps publiques
• Renouveler l'Access Token via le Refresh Token
• Utiliser responseMode: 'fragment' (code dans le #hash)
A ne JAMAIS faire :
• Faire confiance au token sans verifier la signature
• Exposer un client_secret dans du code frontend
• Utiliser le flow Implicit (deprecie depuis OAuth 2.1)
• Envoyer le Refresh Token a une API tierce
